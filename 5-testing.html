Testing

a. Unit testing

To perform our testing, we used tools that are built in the frameworks we are already using.

Smart contracts

Truffle advices the use of Mocha and Chai for unit tests of smart contracts [1]. We have written tests to confirm our smart contracts behaves as expected, and this has helped find some bugs, related to signing a document twice for example. In the last stage, all the tests passed.

b. Integration testing

Because our web app’s functions directly integrates different components (database queries, web3 calls to testRPC…), it was possible to do some integration testing with chai as well. We faced some difficulties because of the client/server strict separation with meteor, and hence some functionalities couldn’t be tested.

e. Feedback from client

Achievement of Objectives*

As the client for this project, I confirm that all Blockdocs objectives were met. Specifically

  • The team have demonstrated the Student and University user interfaces
  • They also showed documents being uploaded, signed and verified, with multiple signatures
  • The administrator’s interface was built though did not have a UI
    • In summary, the team were able to develop a Dual User Interface, with Admin facilities, admirably meeting all “Must Have” and “Should Have” requirements
    • “could” and “would” requirements were also met, through use of open source methods such as MongoDB and Ethereum
    • Specifically, the team should be congratulated on incorporating smart contract functionality, through the use of Ethereum to enable the addition of the uploaded document to the Blockchain
  • My advice to the team would be to continue to test the User Interface, perhaps using colleagues who are not familiar with the project objectives, in order to test how intuitive the interface would be
    • The UI was deliberately clean and minimal and pleasing to the eye, using a very modern colour palette
    • However, one or two small “ease of use” enhancements would be helpful. The team should consider greater separation between the “upload a document” function and the document list. Also, they should consider the position on the screen of the Verify function
    • As a customer I might be inclined to raise a Change Request to allow the Student to view the verification status of the blockchain document
